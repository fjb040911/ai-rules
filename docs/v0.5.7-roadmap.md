# AI-RULES v0.5.7 Roadmap

> This document is the executable roadmap for the next stage of AI-RULES. It is intentionally narrower and more delivery-oriented than the previous brainstorming version.

---

## 1. Roadmap Positioning

The current codebase has already completed several foundational steps:

- rule/config parsing
- config inheritance merge
- lightweight local evidence collection
- audit context generation
- report normalization
- enriched fix prompt generation

That means the next roadmap should not behave like a greenfield plan. It should build on the current architecture rather than introduce too many new abstractions at once.

The key principle for this roadmap is:

**Separate "content expansion" from "engine expansion".**

- Content expansion: more templates, more rules, better i18n, better docs
- Engine expansion: new detect modes, better evidence logic, AST support, severity engine, CLI UX extensions

This separation reduces delivery risk and keeps each version shippable.

---

## 2. Guiding Principles

### 2.1 Deliver In Thin Vertical Slices

Each version should contain:

- some immediately valuable user-facing rules
- only a small number of engine changes
- documentation updates
- test updates

### 2.2 Prefer Low-False-Positive Wins

Before adding advanced rule volume, prioritize:

- security anti-patterns
- pagination / timeout / transaction boundaries
- error swallowing
- direct architecture violations

These yield high value with relatively low ambiguity.

### 2.3 Do Not Over-Promise Full Static Analysis

Until AST support becomes real and tested, features should be described as:

- local evidence
- AI-guided semantic judgment

not as guaranteed deterministic rule enforcement.

### 2.4 Keep Compliance Aspirations Honest

Enterprise/compliance templates are valuable, but they should be framed as:

- enterprise-oriented baseline rules
- compliance-informed patterns

not as full SOC2/GDPR/HIPAA compliance guarantees.

## 3. Workstream Split

The roadmap is split into two parallel workstreams.

### 3.1 Content Track

Focus:

- add high-value rules
- improve templates
- expand i18n
- improve docs

Examples:

- frontend security rules
- backend timeout / pagination / transaction rules
- observability / error-handling rules

### 3.2 Engine Track

Focus:

- add new detect capabilities
- improve evidence quality
- enrich config model
- improve CLI output and workflows

Examples:

- `exceptions`
- `thresholds`
- `count`
- AST parser
- `audit --summary`
- `audit --dry-run`

## 4. Version Strategy

### v0.5.7

Primary goal:

**Expand high-value rules and add low-risk configuration enhancements.**

This version should avoid deep engine leaps.

In scope:

- first batch of high-value security / reliability / performance rules
- `exceptions` support
- `thresholds` support
- README/docs/template updates
- test expansion for new config/rule behavior

Out of scope:

- AST parser
- `call-chain`
- dynamic severity DSL
- interactive fix flow
- enterprise/compliance template rollout

### v0.5.8

Primary goal:

**Improve lightweight detection expressiveness and CLI audit ergonomics.**

In scope:

- first minimal `count` mode
- `audit --summary`
- `audit --dry-run`
- report grouping improvements
- more high-value rules that can reuse the new thresholds

Out of scope:

- full `call-chain`
- free-form dynamic severity conditions
- plugin architecture

### v0.6.0

Primary goal:

**Lay the groundwork for richer structural detection.**

In scope:

- AST parser foundation
- narrow first AST-backed rules
- `call-chain` proof-of-concept
- stronger evidence/report integration

Out of scope:

- broad enterprise/compliance promise
- too many new templates at once

### v0.7.0+

Primary goal:

**Scale the system into a more extensible platform.**

Possible scope:

- enterprise-oriented base templates
- compliance-informed rule packs
- plugin architecture
- richer rule export/docs generation

## 7. Recommended Rule Additions

This is the recommended first wave of new rules. It is intentionally smaller than the earlier brainstorm list.

### 7.1 Frontend

#### Security

- `FE-SEC-101`: unsafe `innerHTML` / `dangerouslySetInnerHTML`
- `FE-SEC-102`: `eval()` / `Function()` dynamic execution
- `FE-SEC-103`: hardcoded secrets or API keys

#### Reliability / Error Handling

- `FE-ERR-101`: third-party risky subtree not protected by Error Boundary
- `FE-OBS-101`: request failure path without standardized error handling

#### Performance

- `FE-PERF-101`: large list without virtualization

### 7.2 Python / FastAPI

#### Security

- `PY-SEC-301`: SQL string concatenation instead of parameterization
- `PY-SEC-302`: weak password hashing algorithm
- `PY-SEC-303`: sensitive information written to logs

#### Reliability / Performance

- `PY-IO-301`: outbound HTTP call without timeout
- `PY-API-305`: list endpoint missing max limit guard
- `PY-PERF-301`: obvious N+1 query pattern in loop

### 7.3 Java / Spring

#### Security

- `JAVA-SEC-301`: SQL concatenation instead of parameterized execution
- `JAVA-SEC-302`: permissive write-path authorization coverage
- `JAVA-SEC-303`: exception handler exposing stack details

#### Reliability / Transactions

- `JAVA-DATA-405`: remote call inside loop
- `JAVA-ERR-301`: business/system exception categories not separated

## 8. Engine Scope Clarification

### 8.1 `exceptions`

This should be implemented before advanced detection expansion.

Recommended initial shape:

```json
{
  "exceptions": {
    "ARCH-101": [
      "src/integrations/**",
      "**/*.stories.tsx"
    ],
    "SEC-*": [
      "__mocks__/**"
    ]
  }
}
```

Why this shape:

- easier for the CLI to consume than free-form prose
- easier to validate
- easier to test
- easier to evolve later

### 8.2 `thresholds`

Recommended initial shape:

```json
{
  "thresholds": {
    "maxFunctionLines": 80,
    "maxParamsCount": 5,
    "maxListLimit": 100,
    "maxInlineComplexity": 15
  }
}
```

Why this should come before dynamic severity DSL:

- easier to merge and validate
- easier to document
- easier to test
- enough for several useful rules

### 8.3 `count`

Recommended first implementation:

- `function-lines`
- `params-count`

Deferred:

- cyclomatic complexity
- nesting depth

Reason:

- lower implementation risk
- easier to explain to users
- easier to validate without pretending to do deep static analysis

### 8.4 `call-chain`

This should not be promised in full before AST infrastructure exists.

Recommended plan:

- v0.5.x: keep as roadmap item only
- v0.6.0: build a narrow proof-of-concept for one language first

### 8.5 Dynamic Severity

Do not implement a free-form conditional DSL yet.

Recommended transitional model:

- fixed severity in rule
- optional severity escalation driven by built-in threshold-aware rules

This avoids introducing a mini rule engine too early.

## 9. CLI UX Recommendations

### Should be prioritized

- `audit --summary`
- `audit --dry-run`

### Should be deferred

- `doctor --fix`
- `fix --interactive`

Reason:

- summary/dry-run are low-risk and high-value
- auto-fixing config is risky
- interactive fix is useful, but not currently the most pressing workflow gap

---

## 10. Template Expansion Strategy

### Near Term

Do not add too many new templates before the engine is a bit stronger.

Recommended near-term template additions:

- one Node.js backend template after the current stacks stabilize

### Later

- `nodejs-express`
- `nodejs-nestjs`
- `go-gin`
- `rust-actix`
- `mobile-react-native`

These are valuable, but they should come after:

- stronger detection foundations
- stable config model
- proven rule authoring workflow

## 12. Definition of Done

For any roadmap item to count as done, it should satisfy:

- implemented in CLI/runtime behavior
- covered by tests when it affects parsing/merging/detection/report behavior
- reflected in template docs/i18n when relevant
- reflected in README or design docs when user-facing
- does not break existing init/audit/fix flows

---

## 13. Summary

This roadmap intentionally narrows the next phase into a more realistic sequence:

- **v0.5.7**: high-value rules + low-risk config features
- **v0.5.8**: lightweight metric detection + audit UX
- **v0.6.0**: AST foundation + structural detection
- **v0.7.0+**: enterprise/platform expansion

That sequencing is designed to keep AI-RULES useful at every stage instead of waiting for a big-bang rewrite.
